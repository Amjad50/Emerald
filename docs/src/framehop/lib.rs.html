1//! # framehop
2//!
3//! Framehop is a stack frame unwinder written in 100% Rust. It produces high quality stacks at high speed, on multiple platforms and architectures, without an expensive pre-processing step for unwind information. This makes it suitable for sampling profilers.
4//!
5//! It currently supports unwinding x86_64 and aarch64, with unwind information formats commonly used on Windows, macOS, Linux and Android.
6//!
7//! You give framehop register values, stack memory and unwind data, and framehop produces a list of return addresses.
8//!
9//! Framehop can be used in the following scenarios:
10//!
11//!  - Live unwinding of a remote process. This is how [`samply`](https://github.com/mstange/samply/) uses it.
12//!  - Offline unwinding from saved registers and stack bytes, even on a different machine, a different OS, or a different CPU architecture.
13//!  - Live unwinding inside the same process. This is currently unproven, but should work as long as you can do heap allocation before sampling, in order to allocate a cache and to update the list of modules. The actual unwinding does not require any heap allocation and should work even inside a signal handler, as long as you use `MustNotAllocateDuringUnwind`.
14//!
15//! As a user of framehop, your responsibilities are the following:
16//!
17//!  - You need to enumerate the modules (libraries) that are loaded in the sampled process ahead of time, or ideally maintain a live list which is updated whenever modules are loaded / unloaded.
18//!  - You need to provide address ranges and unwind section data for those modules.
19//!  - When sampling, you provide the register values and a callback  to read arbitrary stack memory without segfaulting.
20//!  - On aarch64, picking the right bitmask to strip pointer authentication bits from return addresses is up to you.
21//!  - You will need to do symbol resolution yourself, if you want function names. Framehop only produces addresses, it does not do any symbolication.
22//!
23//! In turn, framehop solves the following problems:
24//!
25//!  - It parses a number of different unwind information formats. At the moment, it supports the following:
26//!    - Apple's Compact Unwinding Format, in `__unwind_info` (macOS)
27//!    - DWARF CFI in `.eh_frame` (using `.eh_frame_hdr` as an index, if available)
28//!    - DWARF CFI in `.debug_frame`
29//!    - PE unwind info in `.pdata`, `.rdata` and `.xdata` (for Windows x86_64)
30//!  - It supports correct unwinding even when the program is interrupted inside a function prologue or epilogue. On macOS, it has to analyze assembly instructions in order to do this.
31//!  - On x86_64 and aarch64, it falls back to frame pointer unwinding if it cannot find unwind information for an address.
32//!  - It caches the unwind rule for each address in a fixed-size cache, so that repeated unwinding from the same address is even faster.
33//!  - It generates binary search indexes for unwind information formats which don't have them. Specifically, for `.debug_frame` and for `.eh_frame` without `.eh_frame_hdr`.
34//!  - It does a reasonable job of detecting the end of the stack, so that you can differentiate between properly terminated stacks and prematurely truncated stacks.
35//!
36//! Framehop is not suitable for debuggers or to implement exception handling. Debuggers usually need to recover all register values for every frame whereas framehop only cares about return addresses. And exception handling needs the ability to call destructors, which is also a non-goal for framehop.
37//!
38//! ## Speed
39//!
40//! Framehop achieves high speed in the following ways:
41//!
42//!  1. It only recovers registers which are needed for computing return addresses. On x86_64 that's `rip`, `rsp` and `rbp`, and on aarch64 that's `lr`, `sp` and `fp`. All other registers are not needed - in theory they could be used as inputs to DWARF CFI expressions, but in practice they are not.
43//!  2. It uses zero-copy parsing wherever possible. For example, the bytes in `__unwind_info` are only accessed during unwinding, and the binary search happens right inside the original `__unwind_info` memory. For DWARF unwinding, framehop uses the excellent [`gimli` crate](https://github.com/gimli-rs/gimli/), which was written with performance in mind.
44//!  3. It uses binary search to find the correct unwind rule in all supported unwind information formats. For formats without an built-in index, it creates an index when the module is added.
45//!  4. It caches unwind rules based on address. In practice, the 509-slot cache achieves a hit rate of around 80% on complicated code like Firefox (with the cache being shared across all Firefox processes). When profiling simpler applications, the hit rate is likely much higher.
46//!
47//! Furthermore, adding a module is fast too because framehop only does minimal up-front parsing and processing - really, the only thing it does is to create the index of FDE offsets for `.eh_frame` / `.debug_frame`.
48//!
49//! ## Example
50//!
51//! ```
52//! use core::ops::Range;
53//! use framehop::aarch64::{CacheAarch64, UnwindRegsAarch64, UnwinderAarch64};
54//! use framehop::{ExplicitModuleSectionInfo, FrameAddress, Module};
55//!
56//! let mut cache = CacheAarch64::<_>::new();
57//! let mut unwinder = UnwinderAarch64::new();
58//!
59//! let module = Module::new(
60//!     "mybinary".to_string(),
61//!     0x1003fc000..0x100634000,
62//!     0x1003fc000,
63//!     ExplicitModuleSectionInfo {
64//!         base_svma: 0x100000000,
65//!         text_svma: Some(0x100000b64..0x1001d2d18),
66//!         text: Some(vec![/* __text */]),
67//!         stubs_svma: Some(0x1001d2d18..0x1001d309c),
68//!         stub_helper_svma: Some(0x1001d309c..0x1001d3438),
69//!         got_svma: Some(0x100238000..0x100238010),
70//!         unwind_info: Some(vec![/* __unwind_info */]),
71//!         eh_frame_svma: Some(0x100237f80..0x100237ffc),
72//!         eh_frame: Some(vec![/* __eh_frame */]),
73//!         text_segment_svma: Some(0x1003fc000..0x100634000),
74//!         text_segment: Some(vec![/* __TEXT */]),
75//!         ..Default::default()
76//!     },
77//! );
78//! unwinder.add_module(module);
79//!
80//! let pc = 0x1003fc000 + 0x1292c0;
81//! let lr = 0x1003fc000 + 0xe4830;
82//! let sp = 0x10;
83//! let fp = 0x20;
84//! let stack = [
85//!     1, 2, 3, 4, 0x40, 0x1003fc000 + 0x100dc4,
86//!     5, 6, 0x70, 0x1003fc000 + 0x12ca28,
87//!     7, 8, 9, 10, 0x0, 0x0,
88//! ];
89//! let mut read_stack = |addr| stack.get((addr / 8) as usize).cloned().ok_or(());
90//!
91//! use framehop::Unwinder;
92//! let mut iter = unwinder.iter_frames(
93//!     pc,
94//!     UnwindRegsAarch64::new(lr, sp, fp),
95//!     &mut cache,
96//!     &mut read_stack,
97//! );
98//!
99//! let mut frames = Vec::new();
100//! while let Ok(Some(frame)) = iter.next() {
101//!     frames.push(frame);
102//! }
103//!
104//! assert_eq!(
105//!     frames,
106//!     vec![
107//!         FrameAddress::from_instruction_pointer(0x1003fc000 + 0x1292c0),
108//!         FrameAddress::from_return_address(0x1003fc000 + 0x100dc4).unwrap(),
109//!         FrameAddress::from_return_address(0x1003fc000 + 0x12ca28).unwrap()
110//!     ]
111//! );
112//! ```
